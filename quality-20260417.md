## 4. Performance, Security & Dependencies

### Performance — 62/100 (C+)

**Strengths**
- Ring buffer for fileEvents bounded at 50 (`index.ts:23-31`)
- Skill history capped at 5 entries (`parser.ts:411, 721`)
- Watchers/intervals cleaned on SIGINT/SIGTERM/q/Ctrl+C (`index.ts:135-168`)
- CWD-scoped session lookup avoids scanning all projects (`parser.ts:40-42`)
- Reading `.git/HEAD` directly avoids fork overhead every 2s (`parser.ts:330-344`)

**Weaknesses / risks**
- **Full transcript re-read every 2s**: `readFileSync` + `split('\n')` + per-line `JSON.parse` (`parser.ts:102-112`). For long sessions the transcript can be many MB and is reparsed in full each tick — no mtime guard, no incremental tail
- `findLatestSession` without cwd scans every project dir and `statSync`s every `.jsonl` (`parser.ts:44-57`) — O(N) per refresh in fallback mode
- `loadActiveSessions` does sequential synchronous file reads + `process.kill` for every session every tick (`parser.ts:280-310`)
- All disk I/O is `*Sync` — blocks event loop during render; Bun's async APIs would be a strict improvement
- `new RegExp(...)` recompiled inside `extractRealUserPrompt` per wrapper tag, per call (`parser.ts:387-390`)
- `tasks.find(t => t.id === entry.taskId)` inside the per-line transcript loop is **O(n²)** for sessions with many task updates (`parser.ts:572`)
- Render writes `CLEAR + HIDE_CURSOR + ...` every 2s with no diffing — full screen redraw causes flicker on slow terminals
- `recentFiles` declared but never populated (`types.ts:134`, `parser.ts:87`) — **dead state**

**Hot paths analyzed**
- 2-sec render loop reads entire JSONL transcript synchronously plus 6+ supporting JSON files (subagents, teams, tasks, hook state, sessions, memory, settings cascade) on **every** tick — no caching, no incremental parsing
- File scanner `findLatestSession` at `parser.ts:31-63` — O(projects × files) `statSync` per refresh in fallback path

### Security — 78/100 (B)

**Strengths**
- No `exec`/`spawn`/`eval` in TS code — `.git/HEAD` parsed instead of shelling out
- Every `JSON.parse` wrapped in try/catch
- No hardcoded secrets/credentials
- Bounded ring buffer prevents memory-DoS from spammy file events
- Process-liveness check uses signal 0 (no signal sent) (`parser.ts:199-206`)

**Weaknesses / risks**
- **No sanitization of `process.argv[2]` (sessionId)** before joining into a path (`index.ts:41, 65`). `path.join(fullDir, "${sessionId}.jsonl")` with `sessionId="../../etc/passwd"` resolves outside `~/.claude/projects/<dir>/`. Existence check + `.jsonl` suffix limits exploit to reading `*.jsonl` files within reach — **no `path.relative()` containment guard**
- `findSessionByCwdAndId` (`parser.ts:24-29`) trusts `cwd` from the live `activeSessions` registry (written by other Claude processes) — a malicious `~/.claude/sessions/<pid>.json` could inject `cwd: "../"`
- `cwdToProjectDirName` ↔ `projectDirToRealCwd` not bijective — `_` and `/` both map to `-`, then `-` maps back to `/`. A crafted projectDir could resolve to an arbitrary path passed to `readGitBranch`/`loadEffortLevel`
- `scripts/on-skill-complete.sh` quotes `$INPUT` properly (no shell injection), but no input size limit — a 1GB stdin would buffer
- `readFileSync(transcriptFile)` (`parser.ts:102`) loads whole file — runaway transcript could OOM the monitor; no max-size guard
- File-event paths echoed in UI without ANSI-escape stripping (rare on macOS but possible)

**Input boundaries audited**
- CLI argv → `sessionId` flows directly into `path.join` and `existsSync` with no sanitization, no `..` rejection, no containment check
- Disk reads → JSON parses guarded by try/catch; transcript size unbounded
- Hook stdin → quoted into `node -e` arg in `on-skill-complete.sh:29` (safe from shell injection); resulting JSON is mtime-gated 5 min before use

## 6. Top Issues & Recommendations

### Issue 1 — Zero test coverage on a state-machine parser **[CRITICAL]**
**Affected:** `src/parser.ts`, `src/ui.ts`
**Why it matters:** `processEntry`, `extractRealUserPrompt`, `displayWidth`, `cwdToProjectDirName`, and the skill state machine all have subtle invariants. Regressions here are silent — the UI just renders the wrong thing.
**Recommendation:** Add Bun's built-in test runner. Start with the 5 priority targets in §3. Even a 200-line `parser.test.ts` covering `extractRealUserPrompt` alone would catch most regressions.
```jsonc
// package.json
"scripts": { "test": "bun test", "test:watch": "bun test --watch" }
```

### Issue 2 — Full transcript re-parsed every 2 seconds **[HIGH]**
**Affected:** `parser.ts:65-141, 102-112`
**Why it matters:** Long sessions have multi-MB transcripts. Re-reading + JSON-parsing the entire file every tick wastes CPU and disk I/O even when nothing changed. On a slow disk or large transcript, this can stall the render loop.
**Recommendation:** Track `(transcriptPath, mtimeMs, lastByteOffset, lastState)`. On each tick, `statSync` first; if mtime unchanged, reuse `lastState`. Otherwise, `read` from `lastByteOffset` to EOF and feed only the new bytes through `processEntry`.
```ts
// Sketch
let cache: { mtime: number; offset: number; state: SessionState } | null = null;
const stat = statSync(transcriptFile);
if (cache && cache.mtime === stat.mtimeMs) return cache.state;
const fd = openSync(transcriptFile, 'r');
const bytesToRead = stat.size - (cache?.offset ?? 0);
// ... read from offset, parse new lines, mutate cache.state
```

### Issue 3 — `processEntry` and `render` are god-functions **[HIGH]**
**Affected:** `parser.ts:416-581` (165 lines), `ui.ts:199-499` (300 lines)
**Why it matters:** Both functions concentrate most of the maintenance debt. Adding a feature (new tool source, new render block) means editing the middle of a long, untested function — exactly the high-risk change pattern.
**Recommendation:**
- Split `processEntry` into 5-7 small handlers per role/type (e.g. `handleAssistantMessage`, `handleToolUse`, `handleSkillTag`).
- Extract a `Section` interface in `ui.ts` and replace the 8 inline box renders with `sections.map(s => s.render(state, width))`.

### Issue 4 — Path-traversal & path-asymmetry footguns **[HIGH]**
**Affected:** `index.ts:41, 65`; `parser.ts:13-15, 317-319, 24-29`
**Why it matters:** Two distinct issues compound:
1. `sessionId` from `process.argv[2]` is joined into paths with no `..` rejection or containment check — local path-traversal possible to read `*.jsonl` outside the projects dir.
2. `cwdToProjectDirName` (which maps `_` and `/` both to `-`) is not invertible by `projectDirToRealCwd` (which maps all `-` back to `/`). A user with a path containing `_` (e.g. `/Users/foo_bar/baz`) silently round-trips to the wrong path, breaking git-branch and effort lookups.
**Recommendation:**
```ts
// index.ts — sanitize sessionId
const SESSION_ID_RE = /^[a-zA-Z0-9_-]+$/;
if (sessionId && !SESSION_ID_RE.test(sessionId)) {
  console.error('Invalid sessionId — must match /^[a-zA-Z0-9_-]+$/');
  process.exit(2);
}
// parser.ts — guard the round-trip
function projectDirToRealCwd(dir: string): string {
  // Document the lossy `_` → `-` case OR store the original cwd in a side-channel
  // For now: if no settings/git found at the naive path, log and skip rather than read disk
}
```

### Issue 6 — Sync I/O blocks the event loop on every tick **[MEDIUM]**
**Affected:** `parser.ts` (~19 `*Sync` call sites)
**Why it matters:** With 6+ files read per tick using `readFileSync`/`statSync`/`readdirSync`, a slow disk or large file stalls the entire render loop. Bun's async fs APIs are strictly faster on warm-cache reads too.
**Recommendation:** Convert loaders to `async` and use `Bun.file(path).text()` / `Bun.file(path).json()`. Run independent loaders in parallel: `await Promise.all([loadSubagents(state), loadTeams(state), loadTasks(state), ...])`.

**Recommendation:** Remove or implement.

## 7. Key Insights

★ Insight ─────────────────────────────────────
**The codebase has the readability of a senior engineer's work and the testability of a weekend hack.** Comments explain *why*, types are precise, helpers are well-named — yet `fs.readFileSync` is called inline in 19 places, `Date.now()` is hard-coded, `homedir()` resolves at module load, and `parseTranscript` triggers 7 disk-touching loaders mid-function. The fix is mechanical: extract a `parseTranscriptPure(content, ...)` that does ONLY the `processEntry` loop, then move the loaders to `enrichSessionState(state)`. This single refactor lifts Testability from F to roughly C without touching runtime behavior.
─────────────────────────────────────────────────

★ Insight ─────────────────────────────────────
**The 2-second refresh loop is the single biggest architectural lever.** Every quality dimension touches it: Performance (re-reading full transcripts), Testability (no clock injection), Extensibility (loaders inlined into `parseTranscript`), and even Security (no max-size guard on the readFileSync at `parser.ts:102`). Replacing the "re-parse everything every 2s" model with "read mtime; tail since last offset" would simultaneously cut CPU, force a clean separation of pure parsing vs. I/O, and make the whole parser unit-testable.
─────────────────────────────────────────────────

★ Insight ─────────────────────────────────────
**Asymmetric path encoding (`cwdToProjectDirName` vs. `projectDirToRealCwd`) is exactly the class of bug that survives years of use.** The forward function correctly handles `_` (Claude Code's own escape — see commit `6431f15` "fix/cwd-underscore-conversion"), but the inverse is naive `replace(/-/g, '/')`. A user named `foo_bar` will silently see git/effort lookups against `/Users/foo/bar` instead of `/Users/foo_bar`. The English-only test environment never hits it. A property test asserting `projectDirToRealCwd(cwdToProjectDirName(p)) === p` for a curated set of paths would catch this — and is the cheapest, highest-ROI test to write first.

★ Insight ─────────────────────────────────────
**The 300-line `render()` and the 8-fold copy-paste of box sections are not just a maintainability smell — they're an OCP violation that compounds.** Every new feature ("show MCP server status", "add a Cost section") requires editing the body of a 300-line function in the right vertical position. A `Section[]` array of `{ title, render(state, width) }` objects would let new sections be added without touching `render()` at all, and would make individual sections trivially unit-testable.

★ Insight ─────────────────────────────────────
**Removing chokidar entirely is plausible — and would zero the attack surface.** Bun 1.0+ supports recursive `fs.watch` on macOS and Linux natively. The only chokidar usage (`index.ts:107-122`) is a glob over a fixed list of paths. Replacing it with `fs.watch(CLAUDE_DIR, { recursive: true })` filtered by a few `if (path.includes('/projects/'))` guards would eliminate the single runtime dependency. For a private CLI shipped as a compiled binary, "zero dependencies" is genuinely achievable here.
